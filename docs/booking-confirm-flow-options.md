# 預約「確認」流程改善方案比較

## 背景

目前學生 / 員工建立預約後，狀態是「**待確認**」，必須再由教師（或員工代為）按一次「確認」按鈕，系統才會：
- 建立 Zoom 會議連結
- 發送預約確認通知（Email / LINE）

員工常常需要一次處理一批（例如剛排好一週的固定課），現在只能逐筆點，操作繁瑣。

針對這個痛點，有兩個方向可以選。

---

## 方案 A：拿掉「確認」這一步

### 它怎麼運作
建立預約的當下，系統就直接視為已確認、自動產生 Zoom 連結、自動發通知。沒有「待確認」這個中間狀態。

### 好處
- 使用者體驗最簡：1 步完成，不需要再回來點確認
- 沒有「忘記確認」的可能性

### 代價
- **教師確認流程消失**：目前教師按下確認，是「老師接受這堂課」的隱含紀錄。改成自動確認後，老師等於沒得選，只能事後拒絕（取消預約）
- **補登過去課程變麻煩**：例如老師補登已經上完的課，Zoom 不能建立過去的會議。系統需要額外處理這個例外
- **Zoom 服務暫時不通時，學生會建不出預約**：目前的設計是「先建預約、之後再 confirm 取 Zoom」，Zoom 失敗時不影響建立。改成自動之後，Zoom 失敗 = 整個預約失敗，學生要重來
- **試上課轉正課的規則需要重新設計**：目前有「試上課還沒確認的不能轉正」這條規則，改完之後失去意義
- **代課取消後的處理**：目前代課取消，預約會回到「待確認」讓員工檢查；改完之後會自動再建一次 Zoom，少了人為把關的環節

### 工程量
**約 1.5–2.5 天**（中等規模，跨多個流程要一起調整）

---

## 方案 B：保留兩步流程，加上「批次確認」按鈕

### 它怎麼運作
建立預約後仍然是「待確認」狀態，但員工頁面新增多選功能：勾選一批待確認預約 → 一鍵批次確認。系統自動為每筆建 Zoom、發通知。

如果某幾筆 Zoom 沒建成功（例如帳號池當下被佔滿），那幾筆會留在「待確認」狀態並標記出來，員工事後可以單筆重試。其他成功的不受影響。

### 好處
- **痛點直接解決**：員工 1 個動作處理 100 筆
- **改動範圍小、風險低**：底層流程不變，只是多一個批次入口
- **教師確認流程保留**：老師仍可自己 confirm 自己的課
- **補登過去課程繼續支援**：流程沒動到，原本怎麼做就怎麼做
- **Zoom 失敗有緩衝**：失敗的留在「待確認」可重試，不會整批 fail

### 代價
- 員工仍需要做「確認」這個動作（雖然變成批次）
- 系統內部需要處理「批次中部分失敗」的回應（前端要顯示哪幾筆失敗）

### 工程量
**約 0.5–1 天**

---

## 兩方案對照

| 維度 | 方案 A：拿掉確認 | 方案 B：批次確認 |
|---|---|---|
| 工程量 | 1.5–2.5 天 | 0.5–1 天 |
| 風險 | 中 | 低 |
| 使用者操作步數 | 1 步 | 2 步（但可批量） |
| 教師「接受課程」的訊號 | 消失 | 保留 |
| 補登過去課程 | 需額外處理例外 | 自然支援 |
| Zoom 服務暫時失敗時 | 整筆預約 fail，學生要重來 | 失敗的留在「待確認」可重試 |
| 試上課轉正規則 | 需重新設計 | 不影響 |
| 代課取消後流程 | 自動重建 Zoom，少人為把關 | 員工檢查後再 confirm |

---

## 推薦：方案 B

### 主要理由

1. **痛點直接解決**：員工的核心抱怨是「逐筆點太煩」，批次確認直接命中
2. **風險可控**：不動底層狀態流程，只擴充已有功能，1 天內可完成
3. **保留例外彈性**：補登、Zoom 失敗、代課取消這些邊緣情況不需要重新設計

### 什麼情況才該選方案 A

如果使用者反映的痛點是「為什麼還要再點一次確認、根本是多餘步驟」，那才考慮 A。但即使這樣，也可以用更輕量的「**自動確認**」設定來解：例如「員工建立的非試上預約 → 自動確認」，工程量更小、彈性更大、風險更可控。
